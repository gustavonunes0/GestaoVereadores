# ADR-001 — TramitacaoHistorico substitui tramitacaoJson

**Status:** Aceito | **Data:** 2025-06

## Contexto
`Materia.tramitacaoJson Json @default("[]")` guarda tramitações como array JSON livre. Não há schema definido, não há auditoria de responsável, não há como calcular tempo em cada status.

## Decisão
Criar model `TramitacaoHistorico` append-only. A transaction `(UPDATE materia.status + INSERT tramitacao_historico)` fica em `PrismaMateriaRepository.tramitar()` — nunca no use case.

Campo `tramitacaoJson` continua no schema para compatibilidade com código legado, mas nenhum código novo escreve nele.

## Consequências
- Auditoria completa por responsável, unidade e despacho
- Relatórios de tempo em cada status se tornam possíveis
- LAI e TCE-CE exigem esse nível de rastreabilidade

# ADR-002 — cicloVidaJson substituído por StatusSessao enum + timestamps

**Status:** Aceito | **Data:** 2025-06

## Contexto
`SessaoPlenaria.cicloVidaJson Json?` guarda o ciclo de vida como JSON livre. Impossível verificar estado atual de forma confiável, impossível checar quórum no momento correto.

## Decisão
Criar enum `StatusSessao { AGENDADA ABERTA SUSPENSA ENCERRADA CANCELADA }` com campos de timestamp dedicados: `dataAbertura`, `dataEncerramento`, `dataSuspensao`. Transições validadas em `SessaoPlenaria.podeTransicionarPara()` na entity de domínio.

`cicloVidaJson` continua no schema mas não é escrito por código novo.

# ADR-003 — Contadores de voto são calculados, não armazenados

**Status:** Aceito | **Data:** 2025-06

## Contexto
`Votacao` tem campos `votosSim Int · votosNao Int · abstencoes Int`. São inseridos/atualizados manualmente, podendo divergir do count real de `VotoParlamentar`.

## Decisão
Os campos continuam existindo para performance de leitura, mas só são escritos via `ContagemVotosService.calcular()` que usa `prisma.votoParlamentar.groupBy()`. Nenhum código escreve nesses campos diretamente.

`EncerrarVotacaoUseCase` chama `calcular()`, obtém os valores reais, e os grava em transaction junto com o resultado.

# ADR-004 — VotoParlamentar: FK dual durante migração

**Status:** Aceito | **Data:** 2025-06

## Contexto
`VotoParlamentar` usa `parlamentarId → Parlamentar` (legado PT). Quando a migração de parlamentares for concluída, os votos ficam órfãos.

## Decisão
Adicionar `parliamentarianId String? → Parliamentarian` em `VotoParlamentar`. Durante período de transição, pelo menos uma das duas FKs deve estar preenchida (validação no service). Código novo usa apenas `parliamentarianId`.

Após migração completa dos dados, remover `parlamentarId` via migration futura.

# ADR-005 — StatusNorma é calculado, não armazenado

**Status:** Aceito | **Data:** 2025-06

## Contexto
O ciclo jurídico de uma norma (sanção → veto → promulgação → publicação → vigência → revogação) tem estados deriváveis dos campos de data. Armazenar um campo `status` criaria risco de inconsistência.

## Decisão
`StatusNorma` é um getter na entity de domínio que deriva o status atual dos campos de data:
```
dataRevogacao <= now → REVOGADA
dataVigencia <= now  → VIGENTE
dataPublicacao       → PUBLICADA
dataPromulgacao      → PROMULGADA
dataVeto             → VETADA
dataSancao           → SANCIONADA
(nenhum)             → EM_TRAMITE
```

Não é armazenado no banco. Sempre calculado em runtime.

# ADR-006 — Ato precisa de tenantId

**Status:** Aceito | **Data:** 2025-06

## Contexto
`Ato` é o único model de negócio no schema sem `tenantId`. Portarias, editais e atos da Mesa Diretora são por câmara (tenant), não globais.

## Decisão
Migration M7 adiciona `tenantId String?` (nullable primeiro), popula os dados existentes, depois torna NOT NULL em migration separada.

Todo código novo de `atos-administrativos/` filtra `{ tenantId, isRemoved: false }`.

# ADR-007 — AutorExterno separado de GuestUser

**Status:** Aceito | **Data:** 2025-06

## Contexto
Autores institucionais (Executivo Municipal, OAB, sindicatos, Tribunal de Contas) estão sendo forçados como `GuestUser`. Mas `GuestUser` é uma entidade de identidade (tem login no sistema), não uma entidade legal recorrente.

## Decisão
Criar `AutorExterno` para entidades institucionais e cargos que assinam matérias sem login. `Autor` tem quatro FKs opcionais: `parlamentarId` (legado), `parliamentarianId` (novo), `autorExternoId` (institucional), `guestUserId` (convidado com acesso). `AutorResolverService` valida "exatamente uma FK preenchida".

# ADR-008 — Agenda vinculada a SessaoPlenaria

**Status:** Aceito | **Data:** 2025-06

## Contexto
`AgendaLegislativa` com `tipo = SESSAO` não aponta para `SessaoPlenaria`. Uma agenda que não sabe qual sessão aconteceu é apenas um calendário decorativo.

## Decisão
Adicionar `sessaoPlenariaId String?` em `AgendaLegislativa`. Quando preenchido, `VincularSessaoUseCase` sincroniza `dataInicio`/`dataFim` do evento com os da sessão. Quando sessão encerra, evento agenda atualiza `dataFim`.

Recorrência via ICAL RRULE string — biblioteca `rrule` no backend.
